# Portfolio Evidence Dossier

**Owner:** David Turner  
**Positioning:** Applied AI Workflow & Technical Operations Specialist  
**Purpose:** Help recruiters, hiring managers, collaborators, and buyers distinguish demonstrated evidence from aspiration or unsupported claims.

## Evaluation principle

The portfolio should be judged by the strength of its **artifacts, decisions, verification, and boundaries**:

1. Is there a real problem and identifiable user?
2. Is the workflow coherent and reviewable?
3. Does the source support the public claim?
4. Are important failure modes and limitations documented?
5. Are tests, builds, exports, or proof assets available?
6. Is the work transferable to another reviewer or operator?
7. Are uncertain or future capabilities clearly labeled?

## Evidence classes

### Implemented

Source exists and the functionality can be located in the repository.

### Tested

A documented automated or manual test has been run against a specific version. Testing supports only the tested behavior.

### Deployed

A public deployment reached a ready state and the application identity was checked at a recorded point in time.

### Production-verified

Reserved for a deployed system with its source commit, target environment, critical workflows, data boundaries, and operational controls verified. Most portfolio applications are **not** described this way.

### External validation

Requires independent participants, genuine usage, recorded tasks, observed results, and honest reporting of failures. This remains the largest portfolio evidence gap.
